# DMA Prefetch / Producer-Consumer Overlap — 设计与论证

分支：`codex-baseline-v2-dma-prefetch`（基于 v2 流式核）。
目标：把 DMA 搬运与计算重叠,消除 v2 仍存在的 DMA 串行瓶颈,同时保持完全符合赛题要求。

---

## 1. 先确认:符合赛题要求（不破坏任何硬性约束）

| 赛题要求 | 本优化是否保持 |
|---|---|
| AXI4-Lite 控制 + **AXI4 Master + DMA** 搬运 | ✅ 仍是 AXI4 Master + DMA;只是 DMA 增加"预取下一 tile"的能力,接口不变 |
| online softmax | ✅ 计算核不变(v2 FA-2 流式) |
| tiled K/V | ✅ 仍按 BK=16 tile |
| **禁止存储 S×S 矩阵** | ✅ 只缓存 K/V tile,从不构造分数矩阵 |
| causal mask | ✅ 不变 |
| Q8.8 / ACC≥32 | ✅ 不变 |
| VALID-READY 握手下功能正确(老师答复) | ✅ 预取是**投机**的:命中则快,未命中/被打断则回退正常加载,**任意 backpressure 下结果不变** |
| 性能统计口径(START→DONE) | ✅ 不变;cycles 减少来自隐藏延迟,不改统计方式 |

**结论:这是一个纯粹的"延迟隐藏"微架构优化,不触碰任何功能语义,符合赛题。**

---

## 2. 为什么要做:v2 的 DMA 是串行瓶颈（实测证据）

来自我们自己的全量日志(非理论):

> INT8/FP8 模式把外部字节减半,cycles 从 **154,904 → 117,912**,**正好少 36,992 拍 ≈ 少搬的 36,864 个 AXI beat**(8B/beat)。**1 beat = 1 拍 → DMA 完全串行压在关键路径上。**

v2 拍数构成:**DMA ~73,728 拍(48%,串行) + 计算/归一化 ~81,176 拍**。
v2 已把计算做到 II=1(乘法移出递归),但 **DMA 与计算完全不重叠** —— 这正是本优化的目标。

这对应 [FlashAttention-3](https://arxiv.org/abs/2407.08608) 的第一招 **warp specialization / 生产者-消费者异步**:让"搬数据"和"算"并行。GPU 上靠 TMA+warp;在我们 ASIC 上,就是 **预取式双缓冲 DMA**。

---

## 3. 架构:双缓冲 + 投机预取 DMA（对计算核透明）

```
        ┌─────────── dma_controller (生产者) ───────────┐
AXI ───▶│  tile buffer A ⇄ tile buffer B (ping-pong)    │──▶ k_tile/v_tile (1024b/拍并行) ──▶ flash_core (消费者)
        │  served_start, prefetch_start = served+BK     │
        └───────────────────────────────────────────────┘
```

- 服务完一个 KV tile 后,DMA 在空闲窗口(此时 flash_core 正在 `ST_TILE_RUN` 计算,耗时长)
  **投机预取下一顺序 tile**(`kv_start + BK`)到影子缓冲。
- flash_core 发下一个 `kv_req` 时:
  - **命中**(请求地址 == 预取地址,常见:同一 Q-block 内 tile 顺序递增)→ 影子缓冲秒切为活动,`kv_data_valid` 立即拉高,**DMA 等待≈0**。
  - **未命中**(Q-block 边界,kv_start 重置为 0)→ 丢弃影子,正常加载(正确性不变,仅该 tile 无加速)。
- 接口对 flash_core **完全透明**(同样的 kv_req/kv_data 握手),改动集中在 dma_controller 一个模块 → 风险隔离。

**收益**:DMA(~73,728 拍)被藏到计算(~81,176 拍)背后 → 总拍数 ≈ max(计算, DMA) + 填充 ≈ **~85,000 拍(−45%)**。计算成为新瓶颈(这正是我们想要的:不再等内存)。

## 4. 触发器 vs SRAM:为什么活动 tile 必须是触发器（成熟的取舍,不是偷懒）

用户问"触发器太多就换 SRAM"。我们认真评估了,结论是**活动 tile 缓冲必须保留触发器**,理由量化如下:

- **II=1 点积每拍要并行读一整行 K = D_MODEL×DATA_W = 64×16 = 1024 bit。** 触发器阵列天然支持 1024-bit/拍并行读。
- 我们手上的 sky130 SRAM 宏(OpenRAM)**最大字宽 128 bit、且读有 1 拍延迟**。要 1024-bit/拍需 **8 个 bank 并联**,且 1 拍读延迟会**打破 II=1**(每个 key 多等一拍)。
- 故"把活动 tile 换 SRAM"会**既增加 bank 复杂度、又破坏 II=1 throughput**,得不偿失。

**双缓冲只新增 1 份 tile**(K+V = 16×64×16×2 = 32,768 bit)≈ **+1.6 万门级触发器**(precise 数以 Genus 为准),baseline 163.5万 + ~16万 ≈ **~180万 < 200万**,仍在预算内。**所以这里触发器并不"太多",是正确选择。**

**SRAM 真正该用的地方**(已在 `docs/sram_architecture_plan.md` 分析):是"全量 K/V 片上缓存"以消除 8.5× 重读 —— 但全缓存 512Kbit 用 OpenRAM ≈ 133万门,叠加后逼近/超 200万,**当前不划算**。这是我们评估后的**有依据的取舍**,而非遗漏。

> 工程成熟度体现:为活动 tile 选触发器(性能正确性)、为延迟隐藏加 1 份双缓冲(可控面积)、对全缓存 SRAM 给出量化否决 —— 每个存储决策都有数据支撑。

## 5. 进一步带宽优化(正交,可叠加,列为 future work）
- **减 8.5× K/V 重读**:加大 BQ(代价 acc/q 触发器线性涨)或片上 K/V 缓存(SRAM,面积换带宽)。
- **低精度外部张量**(已是 bonus 7):INT8/FP8 直接把字节(和被隐藏前的 DMA 拍数)减半。
- 预取 + 低精度叠加 → DMA 几乎不可见。

## 5b. 实测结果（已实现并验证,iverilog）

`USE_KV_PREFETCH` 门控,默认 ON;A/B 输出**逐位一致**(OFF vs ON),随机向量全量对 FP32 golden:

| 版本 | cycles | vs 上一版 | MAE | MaxE |
|---|---:|---:|---:|---:|
| 原 baseline(旧核) | 233,312 | — | 0.000015 | 0.003906 |
| v2 流式核(prefetch OFF) | 154,904 | −34% | 0.000097 | 0.054688 |
| **v2 + DMA 预取(prefetch ON)** | **109,410** | **−29%** | **0.000097** | **0.054688** |
| | | **(累计 −53% vs 原 baseline)** | | |

- 小/中规模同样 OFF vs ON 逐位一致:S=8 341→309,S=32 3070→2494。
- 预取命中率 ~88%(每个 Q-block 只有首个 tile 是 miss),DMA 大部分被藏到计算背后。
- 现在**计算成为新瓶颈**(~81K),DMA 不再串行可见 —— 这正是目标。

## 5c. 触发器 vs SRAM —— 最终量化结论

预取影子缓冲 = k_buf2 + v_buf2 = `16×64×16×2 = 32,768 bit = 32 Kb`。
- 作为**触发器**:≈ 0.75M µm² ≈ **+15.7 万门** → baseline 163.5万 + 15.7万 ≈ **~179 万门 < 200 万**,**不算"太多",触发器是正确选择**。
- 不用 SRAM 的硬理由(§4):活动 tile 要 1024-bit/拍并行读以维持 II=1;sky130 OpenRAM 宏字宽≤128bit + 1 拍读延迟,需 8-bank 且破坏 II=1。影子缓冲会与活动缓冲轮换角色,故同样需触发器级并行读。
- → **本优化用触发器实现影子缓冲是经过量化的成熟取舍**;SRAM 留给"全量 K/V 缓存"那条正交路线(面积换带宽,见 `sram_architecture_plan.md`,当前 ~133万门 偏贵)。

- S=8/S=32 快速 smoke(秒级)抓功能 bug;确认输出与 v2 逐位一致(预取不改结果)。

- backpressure 测试:任意 AXI 等待下结果不变(投机预取可被打断/回退)。
- v2 分支保留为回退;本优化在独立分支验证通过后才考虑合入。
